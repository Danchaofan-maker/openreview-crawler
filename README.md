# 写给 Linux 小白的开发环境说明

> 这份说明不讲具体命令，只讲"为什么要这样做"。理解了结构，命令自然就记得住。

---

## 一、你现在运行在哪里

你的电脑上跑着两个操作系统：

```
你的电脑
├── Windows 11         ← 你平时用的桌面系统
└── WSL2 (Ubuntu)     ← 藏在里面的 Linux 系统，我们在这里写代码
```

WSL2 是微软官方提供的 Linux 子系统。它不是虚拟机，更像是 Windows 里面嵌了一个真正的 Linux 内核。你在终端里敲命令，实际上是在操作这个 Linux 系统。

---

## 二、为什么工作目录必须在 Linux 里，不能在 Windows 里

这是最容易犯的错误，也是最重要的事情。

### Windows 的文件在 Linux 眼里是"外来的"

在 WSL2 里，你可以访问 Windows 的文件，路径长这样：

```
/mnt/c/Users/你的用户名/Desktop/my_project/   ← Windows 的桌面
```

但这个路径背后发生的事情是：**每次读写文件，Linux 都要通过一层翻译协议跟 Windows 沟通**。这就像你在说中文，旁边有个翻译帮你转成英文再传过去——能用，但慢，而且有时候会出错。

### 具体会有什么问题

**速度**：在 `/mnt/c/` 里安装 Python 包，比在 Linux 原生目录慢 5 到 10 倍。装一个有几百个文件的库，可能要等几分钟。

**权限系统冲突**：Linux 和 Windows 对文件权限的理解完全不同。Linux 用"谁能读、谁能写、谁能执行"这套系统管理文件，Windows 用另一套。放在 `/mnt/c/` 里的文件，Linux 经常无法正确识别它的权限，导致脚本没法执行，或者虚拟环境里的程序行为异常。

**换行符问题**：Windows 的文本文件每行结尾是 `\r\n`，Linux 是 `\n`。如果你在 Windows 的记事本或某些编辑器里编辑了代码，放到 Linux 里运行，Python 可能会报莫名其妙的语法错误。

### 正确的位置

```
/home/rog/           ← 这是你在 Linux 里的"家"，简写作 ~
└── projects/
    └── openreview-crawler/   ← 项目放这里
```

这里的文件完全由 Linux 管理，没有跨系统翻译，速度正常，权限正常，一切如预期。

## 三、工作目录为什么重要

"工作目录"不只是"文件放在哪个文件夹"这么简单的事。

### 它是所有路径的起点

你的代码里经常需要读取数据文件，比如：

```python
with open("data/raw/iclr_2024_metadata.jsonl") as f:
    ...
```

这个路径是**相对路径**，意思是"从当前工作目录出发，找 data/raw/ 下面的文件"。如果你在错误的目录下运行脚本，它就找不到文件，报错说文件不存在。

工作目录决定了所有相对路径的解释方式。一个项目的所有代码、数据、配置，都应该以同一个根目录为基准来组织，否则团队协作或者换台电脑之后，路径就全乱了。

### 它是虚拟环境的锚点

下面会讲虚拟环境，这里先记住：**虚拟环境是和项目目录绑定的**。你在哪个目录里创建了虚拟环境，就在哪个目录里激活和使用它。跑到别的地方去，用的就是别的环境，装的包可能完全不同。

---

## 四、虚拟环境解决了什么问题

### 想象一个没有虚拟环境的世界

你直接用系统的 Python 装包：

- 项目 A 需要 `requests 2.28`
- 项目 B 需要 `requests 2.31`（API 有变化，不兼容旧版）

你只能装一个版本。装了新的，项目 A 跑不了；装了旧的，项目 B 跑不了。这就是经典的"依赖地狱"。

更危险的是：你把一堆乱七八糟的包装进了系统 Python，系统自带的工具（比如 Ubuntu 的一些管理工具）也依赖 Python，你装错了东西可能把系统搞坏。

### 虚拟环境的解法

虚拟环境就是给每个项目建一个**独立的 Python 小房间**。这个房间里有自己的 Python 解释器和自己的包，和其他项目完全隔离。

```
你的电脑
├── 系统 Python       ← 不动它，保持干净
├── 项目 A 的环境     ← requests 2.28，numpy 1.x
└── 项目 B 的环境     ← requests 2.31，torch 2.x
```

两个项目互不干扰，随时可以切换。

### uv 做了什么

传统方式创建虚拟环境要手动敲命令，手动激活，手动安装，容易忘。

uv 是一个现代工具，它帮你把这些事情自动化了：只要你在项目目录里工作，它就知道该用哪个环境，自动帮你管理依赖。你只需要告诉它"我需要这个包"，它负责装到正确的地方。

## 五、这个项目的文件结构是怎么设计的

```
openreview-crawler/
│
├── scripts/          ← 可以直接运行的主程序
│   ├── openreview_crawler.py      主力爬虫（当前在用）
│   ├── iclr_papers_downloader.py  备选方案 A
│   └── iclr_papers_scraper.py     备选方案 B
│
├── explore/          ← 一次性的探索脚本，用来摸清 API 结构
│   └── probe_*.py                 调试用，不是正式代码
│
├── data/
│   └── raw/          ← 原始数据，只写入，永不手动修改
│       └── *.jsonl
│
├── archive/          ← 已废弃的旧版本，留档备查
│   └── iclr_crawler.py（等）
│
└── pyproject.toml    ← 项目的"身份证"，记录项目名和依赖清单
```

### 为什么这样分

**`scripts/` vs `explore/`**：`scripts/` 里的代码是经过验证、可以重复运行的；`explore/` 里的是摸索过程中写的一次性脚本，逻辑可能不完整，不应该混在一起。

**`data/raw/` 只写入**：这是最重要的原则。原始数据一旦生成就不要碰，它是所有分析的起点。如果你需要清洗数据，把清洗后的版本放到 `data/processed/`，保留原始的。这样出了问题永远可以从头来过。

**`archive/` 而不是直接删除**：代码不确定是否还有用时，先归档。代码删掉了就没了，归档了随时可以翻出来看当时的思路。

**`pyproject.toml`**：这个文件记录了"这个项目需要哪些包"。有了它，换台电脑或者交给别人，一条命令就能还原完全一样的开发环境。没有它，别人拿到你的代码根本不知道要装什么。

---

## 六、一句话总结

> 在 Linux 的家目录（`~/projects/`）下建项目，每个项目一个独立虚拟环境，数据和代码分开放，原始数据只进不改。

这套规范不是规定，是无数人踩坑之后总结出来的。越早养成，越省心。
